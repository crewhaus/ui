# CrewHaus Shape-UI — 0.3.0 Refresh & Panel-System Plan

*Status: proposal for review. Grounds every claim in a read-only survey of the ui repo and factory v0.3.0 (`git show v0.3.0:…`, commit `a609f23`). File:line citations throughout.*

## 0. What we're building

Bring every shape UI up to the 0.3.0 "memory release" and make each one a tasteful, Claude-desktop-style workspace:

1. **A right-side panel system** — chat stays primary; referenced files/plans/wiki/tasks become links in chat that open a viewer on the right. Views: **artifacts, files, background tasks, current focus, current plan (items + statuses), current context, referenced wiki articles, tools in use, skills in use, diff viewer, file upload/attach.** Tasteful and gated per-shape — a voice UI ≠ a cli UI.
2. **Fix cost/token tracking** — cost shows `$0`; audit every displayed number for staleness.
3. **A per-harness settings view** — everything set in the spec (settings, permissions, secrets, tools), editable, with write-back → recompile → **seamless** session continuity across the restart (0.3.0 continuity makes pickup instant).
4. **Each shape gets the views appropriate to its harness.**

The blunt reality up front: **most of this is net-new.** There is no right-side panel system today, no file/artifact/diff viewer, no upload, no settings/spec surface, and the new 0.3.0 memory events are **not on the stream the UI currently reads**. But the substrate is clean and the hooks exist. This plan sequences the work so each phase unblocks the next.

## 1. Ground truth (the architecture we're building on)

**No-build, `window.CH`, one host.** The ui repo is a zero-dependency Bun monorepo. One universal host — `_shared/host.ts` (~905 lines) — spawns the compiled harness and bridges it to the browser over a single WebSocket at `/ws`. 18 shape dirs each ship only `config.json` + a 16-line `index.html` + `app.js`, composing four browser globals: `CH` (`_shared/ui.js`), `CH.events` (`_shared/events.js`), `CH.app` (`_shared/app-kit.js`), and `ui.css`. No framework, no bundler, no state store, **no event replay** on reconnect. A new shared module must be a 5th IIFE under `_shared/` (npm `files` allowlist ships only `_shared/` + each shape's three files).

**Two event surfaces — this is the single most important finding.** Factory has *two* event taxonomies:
- **A — the live trace bus** (`packages/trace-event-bus/src/types.ts`), printed to stdout as JSON Lines under `CREWHAUS_TRACE=json`. This is *all* the UI reads today (`host.ts` `drain()`/`looksLikeEvent()` → `{type:"event"}` over WS).
- **B — the durable session JSONL** at `.crewhaus/sessions/<id>.jsonl` (`packages/event-log/src/index.ts`). A **superset** that includes the 0.3.0 memory kinds.

**The new memory events are on Surface B only.** `plan_update`, `goal_update`, `action_proof`, `wiki_write`, `dream_run`, `context_evicted` are durable-file-only — they never reach the stdout stream the UI consumes. `run_failed`, `cache_rotation`, `compaction`, `model_route`, `sub_agent_*` are on **both**. Implication: the plan/focus/wiki/dream/context panels have **zero live data source today**; we bridge them by having the host **read/watch `.crewhaus/`** (which exposes both human-readable files *and* importable read APIs a Bun host can call directly), keyed off the live `sessionId`+spec name it latches from trace envelopes. The tools that produce those events (`PlanUpdate`, `wiki_write`, …) still appear live as generic `tool_call_start/end`, so "activity" streams live even though structured detail comes from files.

**`config.features[]` is the per-shape view selector — declared but unused.** Every `config.json` has a `features` array (`host.ts:47` documents it as "flags that drive which panels the frontend shows"), e.g. cli `[chat,tools,subagents,mcp,permissions,cost,skills]`, voice `[transcript,turns,voice]`, crew `[roles,handoffs,routing]`. Nothing reads it today (one stale comment in `voice/app.js:421`). This is the ready-made hook for requirement 4.

**Corrected premises — verify before starting:**
- **Local factory is stale.** Local `main` = `40469d6` (v0.2.4); shipped 0.3.0 is `origin/main` `a609f23` + tag `v0.3.0`. The memory packages "look absent" locally — they're real, just unpulled. **Build/test the UI against a factory advanced to v0.3.0.**
- **The UI failure card shipped but isn't on your branch.** `_shared/failure.js` (+ exit-code labels + tests) is on `origin/main` (`cb8958e`, PR #8). The working tree is on `fix/harness-root-cwd` (`3e3855c`), which diverged before it and lacks `origin/main` entirely — plus an uncommitted `host.ts` WIP (runId-less crew events). All three must be reconciled first.
- **Dark-only theme today.** No light theme, no `prefers-color-scheme`, no `[data-theme]` — only the accent is runtime-swappable. Per decision §10.5, light theme lands as an **optional runtime toggle** alongside the accent chooser (net-new but small: a `[data-theme]` layer over the existing CSS custom properties, persisted to `localStorage`).

## 4. Phase 1 — Data plumbing (the foundation)

Everything memory-related depends on getting Surface-B data to the browser and surviving reconnects. Three pieces, all in `host.ts` + a new shared events path.

**1a. The `.crewhaus/` bridge.** Add to the host:
- **Latch identity**: parse `ev.sessionId` (and spec name) from the trace envelope in `drain()`/`pumpStdout` (`host.ts:361/668`) and store it on the `Supervisor` (it's captured by `cli/app.js:106` today but the host never keeps it). This id keys everything below.
- **Read APIs, not re-parsing.** The Bun host can `import` the factory read APIs directly rather than hand-parsing markdown: `createContinuityStore().readFocus()/listPlans()/getActivePlan()/listGoals()/renderHandoff()/listTrash()` (`continuity-store/src/index.ts:467`), `createWikiStore().list()/get()/recall()/stats()` (`wiki-store`), `readDreamState()` (`dream-engine/src/state.ts`), `createSessionStore().list()` + `openEventLog().read({since})` for tailing durable events. `renderHandoff()` is a **ready-made** "current focus / active plan / next-actions" view (`continuity-store/src/handoff.ts`) — preserving the `claimed` vs `proven` distinction.
- **Watch + broadcast.** Watch `.crewhaus/state/<spec>/`, `.crewhaus/wiki/<spec>/`, `.crewhaus/dream/<spec>/`, and the session `.jsonl` (a lightweight `fs.watch`/poll — no chokidar dep, stay zero-dependency); on change, read the relevant store and broadcast a new WS message (e.g. `{type:"memory", surface:"focus"|"plan"|"goals"|"wiki"|"dream"|"handoff", data}`). New panels subscribe via the existing `api.on(...)`.
- **Fix the rooting mismatch.** `.crewhaus/` lives at `harnessRoot` (walked up to `crewhaus.yaml`, `host.ts:133-142`) but `/harness/<path>` is rooted at `harnessDir` — in the `dist/` layout they differ. Add a `.crewhaus/`-aware read/list route rooted at `harnessRoot` (the current `/harness/` route also skips dotfiles, `host.ts:159-161`).

**1b. Event replay on reconnect.** The host keeps no history, so a reload blanks the feed/stats — fatal for the settings-restart flow. Add a bounded ring buffer of recent trace events per run (or re-read the session JSONL on connect) and replay it to a newly-connected WS client after the `state` message. This also makes the feed survive a settings recompile-restart.

**1c. `looksLikeEvent`/renderer extension.** The WIP already widened `looksLikeEvent` for runId-less crew kinds (`host.ts:317-334`) — that's the exact seam. Add renderers to `events.js` `R{}` for the kinds that currently fall to a generic dot card: `run_failed`, `cache_rotation`, `model_route`/`model_tier_route`/`model_failover`, plus any memory kinds we choose to also mirror live. Every shape's feed picks them up for free.

Deliverable: the host exposes live focus/plan/goals/wiki/dream/handoff data + replay; no panels yet, but the data is on the wire.

## 5. Phase 2 — Cost/token truth (the `$0` fix)

**Root cause (triangulated, three-sided):** `cost-tracker` **early-returns without publishing** on a pricing-table miss (`cost-tracker/src/index.ts:165-169`), so a miss suppresses the *entire* `cost_accrual` event — and the UI accrues **both** cost and tokens *only* from `cost_accrual` (`_shared/events.js:262-268`), never from `model_response.usage`. So one unpriced model zeroes both the Cost and Tokens tiles. The pricing table (`pricing.ts`, version `2026-05-08`) tops out at `claude-opus-4-7`/`claude-sonnet-4-6`; a `sonnet-5`-class model misses outright → guaranteed `$0`. (`opus-4-8` matches the `claude-opus-4` base row at a *stale* rate — wrong number, not zero.) Confounder: the UI runs `@crewhaus/*` at npm `latest`, so version skew can zero cost independent of the table.

**The fix (do all three):**
- **UI (P2-a):** in `accrue()`, source `tokensIn/tokensOut` from `model_response.usage` (always present, pricing-independent) so the token tile survives a miss; keep cost from `cost_accrual`. Add an explicit **"unpriced model"** state instead of a silent `$0` (distinguish `null` → "$0.00" from a real 0 → "$0.0000", already in `fmtUsd`).
- **Factory (P2-b = F1+F2):** emit a zero-cost `cost_accrual` carrying real tokens on a miss; refresh the pricing/capabilities/sunset tables to 0.3.0 families. F1 also lights up the existing `alert-watchdog` miss detector for free.
- **Audit the fakes:** cf-worker shapes fabricate tokens as `chars/4` (`cf-worker-*/app.js`), voice/channel track neither — label these honestly (e.g. "estimated") and, where the worker SSE can carry usage, wire the real number.

Deliverable: accurate cost + tokens on every priced model; honest "unpriced"/"estimated" states everywhere else.

## 7. Phase 4 — Settings view + seamless recompile/resume

**The form can be generated from the spec schema.** `@crewhaus/spec` exports the Zod `Spec` discriminated union (`spec/src/index.ts:1793`); the repo already runs `zodToJsonSchema` for tool schemas, so the UI can `zodToJsonSchema(Spec)` to drive a generated form. Top-level blocks per shape are well-defined (agent/model/tools/permissions/mcp_servers/compaction/security/failure_taxonomy/budget/feedback/memory{+wiki+dream}/continuity/thredz/learning/observability + shape-specifics). Caveats to design around: no prebuilt JSON-Schema export (UI adds `zod-to-json-schema`); block schemas are module-private (read from the generated JSON Schema, not per-block imports); `tool_config` values are opaque `unknown` (pass-through editor, or source per-tool schemas from the tool packages); cross-field invariants (`refineModelSelection`, crew/pipeline) live in `parseSpec` → **validate every edit by round-tripping through `parseSpec`.**

**The write path is solved.** `applySpecPatch(yaml, patch)` (`spec-patch/src/index.ts:111`) is **general** (not restricted to `OPTIMIZABLE_PATHS` — that whitelist is the optimizer's separate guard) and **preserves comments + key order** via the YAML CST. `SpecPatch = {target, path: string[], op: "replace"|"add"|"remove", value?, rationale?}`; use `specHasPath()` to choose add vs replace. One limitation: no array-index paths → editing one `permissions.rules[n]` / `failure_taxonomy[n]` / role/step means whole-array `replace` (the same pattern the optimizer uses). It's library-only (no `crewhaus spec set` verb) — the host consumes `@crewhaus/spec-patch` directly.

**Secrets discipline.** The spec stores only `$VAR` refs (`lowerCredential` fails fast on a malformed ref); the **compiled bundle the UI runs reads real values from `.env`** via the host's `loadEnvChain` (`host.ts:111-117`) — *not* `secrets-manager`'s `.crewhaus/secrets/`. So a secret edit = write `$VAR` into the spec (via `applySpecPatch`) **and** write the value into the `.env` the host loads. Per decision §10.2, that value is written to **`harness/.env` by default with a configurable target path** (so a user can point at a parent/shared `.env` or an absolute path outside the repo). The host has **no `.env` writer** today and refuses to *serve* `.env` (`host.ts:854`) — the value-writer is new, carefully-scoped host work; it writes only to the user-chosen path and never ships raw secrets to the browser or into the spec.

**Recompile + seamless resume — the one architecture decision.** The host runs the **prebuilt bundle** (`bun agent.ts`), not `crewhaus run`, so a spec edit has no effect until recompiled, and the compiled bundle **cannot resume** (no `resume` in `target-cli` codegen). Two paths:

- **Path A — keep the compiled-bundle model.** On a settings save: `applySpecPatch` → host calls `compile()` programmatically (`compiler/src/index.ts:130`) → rewrite `harness/` → restart with `CREWHAUS_RESUME=<latched sessionId>`. Requires **F3** (factory: emit resume support in the bundle). Pro: keeps the fast, self-contained "drop a bundle" model. Con: a factory change + an in-host compile step.
- **Path B — switch to the interpreter for live-editing.** Spawn `crewhaus run <spec> [--resume <sessionId>]` instead of `bun agent.ts`. The interpreter **re-reads the spec each start** (recompile is free) and **supports `--resume` natively** — solving *both* problems with **no factory change**. Con: changes the harness model from "compiled bundle" to "spec + `crewhaus` CLI available"; slower startup (parse/lower/install per start).

**Decision (§10.1): both.** The host supports both launch modes and **uses Path B (interpreter) when a spec is present and the settings/live-edit workflow is active**, falling back to the compiled bundle otherwise. This ships seamless edit+resume with zero factory dependency for the MVP; **F3 is a fast-follow** giving compiled bundles the same capability. Either way the host must **latch the sessionId** (Phase 1a) so restart re-attaches the same `.crewhaus/sessions/<id>` — and 0.3.0's continuity (ledger rebuilt from `context_evicted`, plan/goal/memory stores on disk) makes pickup instant. This is exactly where 0.3.0 working memory gets exercised end-to-end.

Deliverable: a generated, validated settings view per shape; edit → patch → recompile/reload → same session resumes with no visible gap.

## 10. Resolved decisions (owner sign-off)

1. **Launch model for settings/resume (§7): BOTH.** Ship Path B (interpreter `crewhaus run`) as the live-edit / settings default — it delivers recompile-free edits + native `--resume` with zero factory dependency — *and* Path A (compiled bundle + factory **F3**) so a compiled harness gets the same seamless resume. The host picks the mode: interpreter when a spec is present and live-editing is active; compiled bundle otherwise. F3 is now a fast-follow (not an MVP blocker, since B ships the capability first).
2. **Secret `.env` location: `harness/.env` by default, with a configurable path.** The settings view writes `$VAR` refs into the spec and the real value into `harness/.env` (which `loadEnvChain` already loads), and exposes a **configurable target path** so a user can point secrets at another location (a shared/parent `.env`, an absolute path outside the repo, etc.). The host still never serves `.env` over HTTP; only the value-writer is new, and it writes to the user-chosen path.
3. **File upload: user-directed, local, never CrewHaus-hosted.** The user chooses *what* files and *where* they land — at minimum a local file on the machine running the agent, or a user-specified destination. CrewHaus hosts nothing in any shared sense; each user owns what they upload and where. The upload control therefore writes to a user-named local path under (or configurably outside) `harnessRoot`, with a clear "this stays on your machine" affordance — no upload endpoint that centralizes files.
4. **Diff viewer (F5) + true context meter (F4): best-judgment sequencing.** Decision: ship the **MVP context meter now** (derived from cumulative `model_response.usage` + compaction/eviction markers — no factory dependency), and treat **F4 (true window-% event)** as an enhancement only if the MVP proves insufficient in practice. The **diff viewer is deferred to a later gated phase**: schedule **F5 (tool-event enrichment)** in factory once the earlier phases land, then build the diff view on it. Neither blocks the panel system.
5. **Light theme: optional runtime toggle, like the accent chooser.** Add a light/dark switch alongside the existing accent-palette chooser (`app-kit.js:47-172`), driven by `[data-theme]` + CSS custom properties and persisted to `localStorage` (mirroring `crewhaus-ui-accent`). Dark stays the default; light is an opt-in toggle, not a `prefers-color-scheme` auto-switch (keeps parity with the deliberate, user-chosen accent model).
6. **Memory panels: host-reads-files (confirmed for real-world use).** The host reads/watches the `.crewhaus/` files/APIs for authoritative *current state* (survives reconnect, gives full state not just deltas, and matches the default `file` backend the standalone-harness UI targets), while the live `tool_use`/`tool_call_*` events drive immediate "activity" feedback. A live trace-bus mirror was rejected: the trace bus is ephemeral (no replay), so panels would still need file reads for initial/reconnect state — mirroring would add factory work without removing the file path. **Backend nuance:** for a `thredz`-backend harness, the wiki/goals panels read via the Thredz path (those live remote), while focus/ledger/plans/dream stay local files — the panel layer abstracts the source behind the same read API.
